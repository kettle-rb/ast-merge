Key Principles Discovered

1. TreeHaver is for Grammar-Based Parsing

Use TreeHaver when:

  • ‚úÖ Language uses tree-sitter grammars
  • ‚úÖ Grammar needs to be discovered/loaded dynamically
  • ‚úÖ Multiple backends might be used (MRI, Rust, FFI, Java, Citrus)
  • ‚úÖ Security validation of shared libraries is needed
  • ‚úÖ Platform-specific handling is required

Don‚Äôt use TreeHaver when:

  • ‚ùå Language has a native Ruby parser (Prism, Psych, RBS)
  • ‚ùå Only one parser implementation exists
  • ‚ùå No grammar file needs to be loaded
  • ‚ùå Parser is part of Ruby stdlib

2. Merge Logic vs Parsing Logic

TreeHaver provides:

  • Parsing API abstraction
  • Grammar discovery and loading
  • Backend switching capability
  • Node access normalization

TreeHaver does NOT provide:

  • Freeze block detection
  • Comment tracking for merging
  • Signature generation
  • Conflict resolution
  • Merge strategies

The *-merge gems provide the merge intelligence.

3. Duplication Can Be Appropriate

Not all similar code is inappropriate duplication:

  • ‚úÖ Appropriate: Domain-specific implementations (e.g., YAML signature generation vs Ruby signature generation)
  • ‚úÖ Appropriate: Similar patterns with different constraints (e.g., freeze block detection)
  • ‚ùå Inappropriate: Identical hardcoded paths across multiple gems
  • ‚ùå Inappropriate: Copy-pasted grammar loading logic
